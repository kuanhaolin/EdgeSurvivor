## 1. 專案概述

### 1.1 專案名稱與目標

**專案名稱：** EdgeSurvivor（邊緣人神器）- 旅伴媒合與活動管理平台

**專案目標：**
- 建立一個專注於「旅伴與活動媒合」的社交平台
- 解決獨自旅行者尋找志同道合旅伴的痛點
- 提供活動創建、智能媒合、即時聊天、行程討論、費用分攤等完整功能
- 讓使用者能安全、便捷地找到理想的同行夥伴

### 1.2 專案範圍

**包含範圍（MVP 已完成）：**
- ✅ 使用者認證系統（註冊、登入、JWT、OAuth2）
- ✅ 個人資料管理（頭像、Bio、興趣、社群連結）
- ✅ 活動管理（建立、搜尋、篩選、參與、審核）
- ✅ 活動討論串（即時交流）
- ✅ 媒合系統（申請、接受、拒絕）
- ✅ 即時聊天室（Socket.IO）
- ✅ 費用分攤系統（記錄、計算、結算）
- ✅ 活動評價系統（互評）
- ✅ 檔案上傳服務（圖片、頭像、封面）
- ✅ 兩步驟驗證（2FA）
- ✅ Email 驗證與密碼重設

**不包含範圍（Phase 2 以後）：**
- 推播通知（Email/App）
- 地圖整合（Google Maps）
- 天氣資訊整合
- AI 推薦演算法
- 多語系支援
- 原生行動 App
- 付款整合
- 管理後台

### 1.3 專案約束

**時間約束：**
- MVP 開發週期：3-4 個月（已完成）
- Beta 測試期：1 個月（當前階段）
- 正式上線：預計 1 個月後

**預算約束：**
- 開發成本：開源技術棧，無授權費用
- 主機成本：初期使用低成本 VPS（月費 $10-20 USD）
- 人力成本：小團隊開發（1-3 人）

**技術約束：**
- 後端：Python Flask 3.x
- 前端：Vue 3 + Vite
- 資料庫：MariaDB 10.11 / MySQL 8.x
- 部署：Docker + Docker Compose

### 1.4 專案假設

- 目標用戶具備基本的網路使用能力
- 台灣旅遊市場持續成長，獨旅需求穩定
- 使用者信任線上媒合平台
- Email 仍是主要的通知管道
- 大部分用戶會遵守社群規範

## 3. 管理流程

### 3.1 專案啟動流程

1. **需求收集與分析**
   - 市場調查與競品分析
   - 目標用戶訪談（20 位潛在用戶）
   - 痛點識別與需求整理

2. **專案可行性評估**
   - 技術可行性：✅ 技術棧成熟穩定
   - 市場可行性：✅ 獨旅市場年產值 4,000 億台幣
   - 經濟可行性：✅ 開源方案，低成本啟動

3. **專案章程制定**
   - 專案簡報 (Project Brief) 完成
   - 產品需求文件 (PRD) 完成
   - 獲得專案發起人批准

### 3.2 專案規劃流程

1. **範圍規劃**
   - 定義 MVP 核心功能
   - 識別 Phase 2 延後功能
   - 建立需求優先級矩陣

2. **時程規劃**
   - WBS 工作分解
   - 里程碑設定
   - 甘特圖繪製

3. **資源規劃**
   - 人力資源分配
   - 技術資源準備
   - 預算編列

### 3.3 專案執行與監控流程

**每日站立會議 (Daily Standup)：**
- 時間：每日早上 10:00，15 分鐘
- 內容：
  - 昨天完成了什麼？
  - 今天計畫做什麼？
  - 遇到什麼阻礙？

**每週進度會議 (Weekly Review)：**
- 時間：每週五下午 3:00，1 小時
- 內容：
  - 本週完成功能展示
  - 下週工作規劃
  - 風險與問題討論

**Sprint 迭代週期：**
- 週期：2 週一個 Sprint
- 流程：Sprint Planning → Development → Sprint Review → Retrospective

### 3.4 變更管理流程

1. **變更請求提出**
   - 任何團隊成員或利害關係人可提出變更請求
   - 填寫變更請求表單（包含變更描述、理由、影響）

2. **變更影響評估**
   - PM 與 Architect 評估變更影響（時程、成本、範圍）
   - 評估風險與效益

3. **變更決策**
   - 小型變更：PM 決定
   - 中型變更：開發團隊討論決定
   - 重大變更：專案發起人批准

4. **變更實施**
   - 更新專案文件
   - 通知相關人員
   - 追蹤變更執行

## 4. 技術流程

### 4.1 開發流程

**採用 Agile 敏捷開發方法論**

1. **功能開發流程**
   ```
   需求分析 → 設計 → 開發 → 測試 → Code Review → 合併 → 部署
   ```

2. **Git 分支策略**
   - `main` - 生產環境（穩定版本）
   - `develop` - 開發環境（整合分支）
   - `feature/*` - 功能分支
   - `bugfix/*` - Bug 修復分支
   - `hotfix/*` - 緊急修復分支

3. **程式碼規範**
   - 後端：PEP 8 (Python)
   - 前端：ESLint + Prettier (Vue 3)
   - 命名規範：snake_case (後端)、camelCase (前端)

### 4.2 測試流程

**測試策略：**

| 測試類型 | 負責人 | 時機 | 工具 |
|---------|--------|------|------|
| 單元測試 | 開發者 | 開發階段 | Pytest (後端)、Vitest (前端) |
| 整合測試 | 開發者 | 功能完成後 | Pytest + API Client |
| 系統測試 | QA/種子用戶 | Sprint 結束後 | 手動測試 |
| 使用者驗收測試 (UAT) | 種子用戶 | Beta 階段 | 真實使用場景 |

**測試環境：**
- 開發環境：本地 Docker
- 測試環境：Docker Compose（與生產環境一致）
- 生產環境：VPS + Nginx + Docker

### 4.3 部署流程

**部署策略：手動部署（MVP 階段）**

1. **開發環境部署**
   ```bash
   docker-compose up -d
   ```

2. **測試環境部署**
   - 合併至 `develop` 分支
   - 執行自動化測試
   - 部署至測試伺服器

3. **生產環境部署**
   - 合併至 `main` 分支
   - 建立 Git Tag（版本號）
   - 執行部署腳本
   - 健康檢查
   - 監控系統指標

**部署檢查清單：**
- [ ] 資料庫遷移完成
- [ ] 環境變數配置正確
- [ ] 靜態檔案上傳
- [ ] SSL 憑證有效
- [ ] API 健康檢查通過
- [ ] 前端頁面正常顯示
- [ ] Socket.IO 連線正常

## 8. 風險管理

### 8.1 風險識別與評估

| 風險 ID | 風險描述 | 影響 | 機率 | 風險等級 | 負責人 |
|---------|---------|------|------|---------|--------|
| **R1** | 安全風險：陌生人媒合可能產生人身安全疑慮 | 高 | 中 | **高** | PM |
| **R2** | 冷啟動問題：平台初期活動數量少 | 高 | 高 | **高** | PM |
| **R3** | 競爭風險：Facebook 推出類似功能 | 中 | 低 | **中** | PM |
| **R4** | 法規風險：可能被視為旅行社需符合法規 | 中 | 低 | **中** | PM |
| **R5** | 技術風險：流量暴增導致伺服器負載過高 | 高 | 低 | **中** | Architect |
| **R6** | 商業模式風險：免費平台難以維持營運 | 中 | 中 | **中** | PM |
| **R7** | 人力風險：關鍵開發人員離職 | 高 | 低 | **中** | PM |
| **R8** | 資料外洩：使用者個資遭駭客攻擊 | 高 | 低 | **中** | Architect |

**風險等級定義：**
- **高**：影響大且機率高，需立即應對
- **中**：影響或機率其中之一為中等，需監控
- **低**：影響小且機率低，持續觀察

### 8.2 風險應對策略

**R1: 安全風險 - 陌生人媒合安全疑慮**
- **緩解措施：**
  - ✅ 身分驗證機制（Email 驗證、兩步驟驗證）
  - ✅ 活動評價系統（累積信用）
  - 📋 安全使用指南（Beta 階段新增）
  - 📋 舉報與封鎖機制（Phase 2）
- **應急計畫：**
  - 若發生安全事件，立即下架相關活動
  - 提供客服支援與協助
  - 與執法單位合作

**R2: 冷啟動問題 - 初期活動數量少**
- **緩解措施：**
  - 🔄 招募 50-100 位種子用戶（進行中）
  - 📋 官方發起示範活動（Beta 階段）
  - 📋 社群經營（Facebook 粉專、Instagram）
  - 📋 KOL 合作推廣
- **應急計畫：**
  - 提供早鳥用戶獎勵
  - 舉辦線下見面會

**R5: 技術風險 - 伺服器負載過高**
- **緩解措施：**
  - ✅ Docker 容器化部署（易於擴展）
  - 📋 監控系統（Uptime、Response Time）
  - 📋 CDN 加速靜態資源
  - 📋 資料庫讀寫分離（若需要）
- **應急計畫：**
  - 垂直擴展（升級 VPS 方案）
  - 水平擴展（增加伺服器節點）
  - 暫時限制新用戶註冊

**R8: 資料外洩風險**
- **緩解措施：**
  - ✅ 密碼加密（scrypt）
  - ✅ HTTPS 強制加密
  - ✅ SQL 注入防護（ORM）
  - 📋 定期安全審計
  - 📋 資料庫備份機制
- **應急計畫：**
  - 立即通知受影響用戶
  - 強制重設密碼
  - 公開事件處理報告

### 8.3 風險監控

- **風險回顧頻率：** 每週進度會議
- **風險負責人：** 專案經理
- **風險報告：** 每月風險報告（若有高風險事件）

## 9. 品質管理

### 9.1 品質標準

**功能品質標準：**
- 所有 MVP 核心功能正常運作
- 無重大 Bug（Severity: Critical）
- API 回應時間 < 500ms
- 系統可用性 > 99%（Beta 階段）

**程式碼品質標準：**
- 後端：符合 PEP 8 規範
- 前端：通過 ESLint 檢查
- 程式碼覆蓋率 > 60%（單元測試）
- Code Review 通過率 100%

**使用者體驗品質標準：**
- 註冊流程 < 3 分鐘完成
- 發布活動 < 5 分鐘完成
- 搜尋結果 < 2 秒顯示
- 聊天訊息即時送達（< 1 秒）

### 9.2 品質保證活動

**程式碼審查（Code Review）：**
- 頻率：每個功能合併前
- 審查者：技術架構師 / 資深開發者
- 檢查項目：
  - 程式碼邏輯正確性
  - 符合編碼規範
  - 無安全漏洞
  - 效能考量

**測試活動：**
| 測試類型 | 頻率 | 負責人 | 目標覆蓋率 |
|---------|------|--------|-----------|
| 單元測試 | 每次提交 | 開發者 | > 60% |
| 整合測試 | 每個 Sprint | 開發者 | 關鍵 API |
| 系統測試 | Sprint 結束 | QA | 所有功能 |
| UAT | Beta 階段 | 種子用戶 | 真實場景 |

**品質檢查清單（Release Checklist）：**
- [ ] 所有功能通過測試
- [ ] 無 Critical/High Bug
- [ ] 效能指標達標
- [ ] 安全審查通過
- [ ] 文件更新完成
- [ ] 備份機制就緒

### 9.3 缺陷管理

**Bug 嚴重性分級：**
| 等級 | 描述 | 處理時間 |
|------|------|----------|
| **Critical** | 系統無法使用、資料遺失、安全漏洞 | 24 小時內修復 |
| **High** | 主要功能無法使用 | 3 天內修復 |
| **Medium** | 功能部分異常、影響使用體驗 | 1 週內修復 |
| **Low** | 介面顯示問題、非關鍵功能 | 下個 Sprint 修復 |

**Bug 追蹤工具：** GitHub Issues

**Bug 處理流程：**
1. 回報 → 2. 分類 → 3. 指派 → 4. 修復 → 5. 驗證 → 6. 關閉

## 10. 配置管理

### 10.1 版本控制

**版本控制系統：** Git  
**代碼庫：** GitHub（https://github.com/BoBoJhong/EdgeSurvivor）

**分支策略（Git Flow）：**
```
main（生產環境）
  ├── develop（開發環境）
  │    ├── feature/user-auth
  │    ├── feature/activity-mgmt
  │    └── feature/chat-room
  ├── bugfix/fix-login-error
  └── hotfix/critical-security-patch
```

**提交訊息規範（Conventional Commits）：**
```
<type>(<scope>): <subject>

type: feat, fix, docs, style, refactor, test, chore
scope: auth, activity, chat, expense, etc.

範例：
feat(auth): add Google OAuth2 login
fix(chat): resolve message not sending issue
docs(readme): update deployment guide
```

### 10.2 發布管理

**版本號規則（Semantic Versioning）：**
```
MAJOR.MINOR.PATCH

1.0.0 - MVP 正式上線
1.1.0 - Phase 2 功能增強
1.0.1 - Bug 修復
```

**發布流程：**
1. 建立 Release Branch（`release/v1.0.0`）
2. 執行完整測試
3. 更新版本號與 CHANGELOG
4. 合併至 `main` 並打 Tag
5. 部署至生產環境
6. 回合併至 `develop`

### 10.3 配置項管理

**環境配置：**
- `.env.development` - 開發環境
- `.env.test` - 測試環境
- `.env.production` - 生產環境

**敏感資訊管理：**
- 資料庫密碼
- JWT Secret Key
- Google OAuth2 Credentials
- Email SMTP 密碼

**管理方式：**
- 使用環境變數
- 不提交至 Git（`.gitignore`）
- 使用 Secret Management 服務（未來）
